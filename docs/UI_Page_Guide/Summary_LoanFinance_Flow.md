## ภาพรวมระบบ

ระบบสินเชื่อแบ่งเป็น 2 ส่วนหลัก:

| ส่วน | ชื่อ | ความหมาย |
|-----|------|---------|
| 🏦 ส่วนที่ 1 | สินเชื่อไฟแนนซ์ภายนอก | บันทึกประวัติการขอสินเชื่อกับแต่ละบริษัทไฟแนนซ์ ค้นหาด้วยเบอร์/บัตรปชช. |
| 🏪 ส่วนที่ 2 | สินเชื่อร้าน | ร้านให้สินเชื่อเอง กำหนดยอด/ดอก/งวดเอง ผ่อนตรงกับร้าน |

## 5 ตารางหลักในระบบ

| ตาราง | ส่วน | คำอธิบาย |
|-------|-----|---------|
| finance_companies | ไฟแนนซ์ | Master data บริษัทสินเชื่อภายนอก เช่น กรุงไทยลีสซิ่ง, ไทยซัมมิท, กรุงศรีออโต้ |
| loan_applications | ไฟแนนซ์ | ประวัติการขอสินเชื่อแต่ละครั้ง — ค้นด้วยเบอร์/บัตรปชช. |
| loan_application_guarantors | ไฟแนนซ์ | ผู้ค้ำประกัน — ค้นหาว่าเคยค้ำให้ใครที่เจ้าไหน |
| store_loans | ร้าน | สัญญาสินเชื่อร้าน — กำหนดยอด/ดอก/งวดเอง |
| store_loan_payments | ร้าน | บันทึกการชำระงวดสินเชื่อร้านแต่ละงวด |

## การค้นหา (Elasticsearch)

**★ ใช้ Elasticsearch ค้นหา — ห้ามใช้ MySQL LIKE (ช้าเกินไป)**

| ค้นหาด้วย | ผลที่ได้ | หมายเหตุ |
|---------|--------|---------|
| เบอร์โทร (ผู้ขอ) | ประวัติขอสินเชื่อทุกรายการ — เจ้าไหน ผ่าน/ไม่ผ่าน | INDEX บน applicant_phone |
| บัตรปชช. (ผู้ขอ) | ประวัติขอสินเชื่อทุกรายการ | INDEX บน applicant_id_card — ค้นแม้เปลี่ยนเบอร์ |
| เบอร์โทร (ผู้ค้ำ) | เคยค้ำให้ใคร เจ้าไหน ผ่านหรือไม่ | INDEX บน guarantor_phone |
| บัตรปชช. (ผู้ค้ำ) | เคยค้ำให้ใคร เจ้าไหน | INDEX บน guarantor_id_card |

**Sync Policy:** create/update → index เข้า Elasticsearch ทันที (real-time sync)

**Search endpoint:** `GET /loans/search` — คืน as_applicant + as_guarantor + store_loans ใน response เดียว

## Scenario A — บันทึกการขอสินเชื่อไฟแนนซ์ภายนอก

```
ค้นหาประวัติ (เบอร์/บัตร) → กรอกแบบฟอร์ม (ผู้ขอ + ผู้ค้ำ) → ส่งเรื่อง (pending) → อัปเดตผล (approved/rejected)
```

| หัวข้อ | รายละเอียด |
|--------|-----------|
| 1. ค้นหาประวัติก่อน | ค้นด้วยเบอร์หรือบัตรปชช. → แสดงประวัติทั้งหมดของคนนั้น (ขอเอง + ค้ำ + สินเชื่อร้าน) |
| 2. กรอกแบบฟอร์ม | บันทึก: ชื่อผู้ขอ, เบอร์, บัตรปชช., บริษัทไฟแนนซ์, ยอดที่ขอ, วันที่ยื่น ผูก customer_id (ถ้าอยู่ในระบบ) / quotation_id หรือ invoice_id ถ้ามี |
| 3. เพิ่มผู้ค้ำ | กรอก: ชื่อผู้ค้ำ, เบอร์, บัตรปชช., ความสัมพันธ์ — เพิ่มได้หลายคน |
| 4. ส่งเรื่อง | อัปเดต status=pending — บันทึกว่ายื่นกับไฟแนนซ์แล้ว |
| 5. อัปเดตผล | approved: บันทึก amount_approved + approved_date │ rejected: บันทึก note เหตุผล |

**loan_application Status:**

| status | ความหมาย |
|--------|---------|
| pending | ยื่นเรื่องแล้ว รอผล |
| approved | ไฟแนนซ์อนุมัติ |
| rejected | ไฟแนนซ์ปฏิเสธ |
| cancelled | ยกเลิกคำขอ |

**ตัวอย่าง Scenario A — นายสมชาย ขอสินเชื่อกรุงศรีออโต้:**

| ขั้น | ข้อมูล | รายละเอียด |
|-----|--------|-----------|
| ① ค้นหาประวัติ | 081-234-5678 | พบ 2 รายการ: กรุงไทยลีสซิ่ง (approved 2024), ไทยซัมมิท (rejected 2025) |
| ② บันทึกใหม่ | LA-2026-0031 | ชื่อ: นายสมชาย │ เบอร์: 081-234-5678 │ บัตรปชช: 1100XXXXXXXXX |
| ③ ไฟแนนซ์ | กรุงศรีออโต้ | amount_requested=45,000 │ applied_date=09/04/2026 |
| ④ ผู้ค้ำ | นางสาวสุดา (ภรรยา) | เบอร์: 089-XXX-XXXX │ บัตรปชช: 1200XXXXXXXXX |
| ⑤ ผล | 3 วันต่อมา | status=approved │ amount_approved=45,000 │ approved_date=12/04/2026 |

## Scenario B — ค้นหาประวัติสินเชื่อ

```
กรอกเบอร์/บัตร → Elasticsearch ค้น loan_apps + guarantors + store_loans → แสดงประวัติครบ → วิเคราะห์โอกาส
```

**ผลการค้นหา — แสดงอะไรบ้าง:**

| ส่วน | รายละเอียดที่แสดง |
|-----|----------------|
| ประวัติการขอ (as_applicant) | เจ้าไฟแนนซ์, วันที่ยื่น, ยอดขอ, ยอดอนุมัติ, สถานะ, note |
| ประวัติค้ำประกัน (as_guarantor) | ค้ำให้ชื่อใคร, เจ้าไหน, วันที่, ผลอนุมัติ, ความสัมพันธ์ |
| สินเชื่อร้าน (store_loans) | เคยกู้กับร้านไหม ยอดเท่าไหร่ สถานะ (active/completed/overdue) |

**ความต่างของ เบอร์ vs บัตรปชช.:**

| หัวข้อ | รายละเอียด |
|--------|-----------|
| ทำไมต้องแยก | ลูกค้าอาจเปลี่ยนเบอร์โทร แต่บัตรปชช. ไม่เปลี่ยน → ค้นบัตรปชช. ได้ประวัติครบกว่า |
| เบอร์โทร | ค้นหาเร็ว เหมาะใช้ตอนลูกค้ามาหน้าร้าน — อาจไม่ครบถ้าเปลี่ยนเบอร์ |
| บัตรปชช. | ค้นหาครบที่สุด — ใช้ยืนยันตัวตนเมื่อต้องการประวัติทั้งหมด |

**ตัวอย่าง Scenario B — ค้นประวัตินายสมชาย:**

| ขั้น | ข้อมูล | รายละเอียด |
|-----|--------|-----------|
| ① ค้นเบอร์ | 081-234-5678 | ระบบแสดงประวัติ: ขอเอง 3 รายการ, ค้ำให้คนอื่น 1 รายการ |
| ② as_applicant | กรุงไทย 2024 | approved 40,000 บาท ✅ |
| ③ as_applicant | ไทยซัมมิท 2025 | rejected ❌ |
| ④ as_applicant | กรุงศรี 2026 | pending ⏳ |
| ⑤ as_guarantor | กรุงไทย 2024 | ค้ำให้นายวิชัย — approved ✅ |
| ⑥ วิเคราะห์ | เซลแนะนำ | ลองกรุงศรีก่อน เพราะไทยซัมมิทเคย reject |

## Scenario C — สินเชื่อร้าน (Store Loan)

```
ตกลงเงื่อนไข (ยอด/ดอก/งวด) → สร้างสัญญา → เงินดาวน์ (ถ้ามี) → ตัดยอดรายงวด → ครบงวดสุดท้าย → completed
```

| หัวข้อ | รายละเอียด |
|--------|-----------|
| จุดเริ่มต้น | ลูกค้าตกลงผ่อนกับร้าน — ไม่ผ่านบริษัทไฟแนนซ์ภายนอก |
| เลขที่สัญญา | SL-{ปี}-{เลขลำดับ} เช่น SL-2026-0005 |
| เงื่อนไข | ร้านกำหนดเอง: ยอดกู้ (principal), ดอกเบี้ย % ต่อปี, จำนวนงวด, ค่างวดต่อเดือน |
| เงินดาวน์ | ★ เก็บ down_payment field — หักจากราคาก่อนคำนวณ principal: principal = ราคาสินค้า − down_payment |
| คำนวณค่างวด | PMT = P × r(1+r)^n / ((1+r)^n − 1) หรือกรอก monthly_payment เองได้ |
| ★ principal | **principal = ราคาสินค้า − down_payment** → ยอดที่นำไปคำนวณ PMT |
| ตัดยอดรายงวด | บันทึก store_loan_payment ทุกครั้งที่ลูกค้ามาจ่าย (งวดที่, จำนวน, วิธีชำระ, วันที่) |
| ปิดสัญญา | ชำระครบงวดสุดท้าย → status=completed อัตโนมัติ |

**Store Loan Status:**

| status | ความหมาย | เปลี่ยนเมื่อ |
|--------|---------|-----------|
| active | อยู่ระหว่างผ่อนชำระ | สร้างสัญญา |
| completed | ชำระครบทุกงวดแล้ว | ชำระงวดสุดท้าย |
| overdue | ค้างชำระ เกิน due date | Job รายวัน (01:00) |
| cancelled | ยกเลิกสัญญา | ผู้จัดการยกเลิก |

**Overdue Check Job:** ทำงานทุกวัน เวลา 01:00 — store_loans งวดถึงกำหนดแต่ไม่มี payment → set status=overdue + notification

**ตัวอย่าง Scenario C — ผ่อน Honda MSX 12 งวด:**

| ขั้น | ข้อมูล | รายละเอียด |
|-----|--------|-----------|
| ① ตกลง | Honda MSX | ราคา 72,000 │ ดาวน์ 12,000 (★ down_payment) │ เหลือผ่อน 60,000 │ ดอก 12%/ปี │ 12 งวด |
| ② คำนวณ | PMT formula | PMT(60,000, 1%/เดือน, 12 งวด) = **5,330 บาท/งวด** |
| ③ สัญญา | SL-2026-0005 | principal=60,000 │ interest_rate=12.00 │ installments=12 │ monthly_payment=5,330 │ start_date=01/05/2026 |
| ④ งวดที่ 1 | 01/05/2026 | amount=5,330 │ method=transfer │ บันทึก store_loan_payment |
| ⑤ งวด 2–11 | ทุกเดือน | ตัดยอดทุกเดือน — status=active ตลอด |
| ⑥ งวดที่ 12 | 01/04/2027 | ชำระครบ → status=completed อัตโนมัติ |

## สรุปเปรียบเทียบ 2 ระบบ

| หัวข้อ | สินเชื่อไฟแนนซ์ภายนอก | สินเชื่อร้าน |
|--------|---------------------|-----------|
| วัตถุประสงค์ | บันทึกประวัติ + ค้นหา | สร้างสัญญาผ่อนกับร้าน |
| ใครให้เงิน | บริษัทไฟแนนซ์ | ร้าน |
| กำหนดดอกเบี้ย | ไฟแนนซ์กำหนด | ร้านกำหนดเอง |
| ค้นหาด้วย | เบอร์, บัตรปชช. (ผู้กู้ + ผู้ค้ำ) | เบอร์, ชื่อ, customer_id |
| status | pending, approved, rejected, cancelled | active, completed, overdue, cancelled |
| ผู้ค้ำ | บันทึกได้ — ค้นหาย้อนหลังได้ | ไม่มี |
| เชื่อมการขาย | ผูก quotation_id / invoice_id ได้ | ผูก invoice_id ได้ |
| Search engine | Elasticsearch (multi-index) | Elasticsearch |

## API Endpoints

| Method | Endpoint | คำอธิบาย |
|--------|----------|---------|
| GET | /loan-applications | รายการขอสินเชื่อไฟแนนซ์ |
| POST | /loan-applications | บันทึกการขอสินเชื่อใหม่ |
| GET | /loan-applications/:id | รายละเอียดคำขอ |
| PUT | /loan-applications/:id | แก้ไขคำขอ (pending เท่านั้น) |
| PATCH | /loan-applications/:id/status | อัปเดตผล (approved/rejected) |
| GET | /loans/search | Elasticsearch ค้น phone/id_card — คืน applicant + guarantor + store_loans |
| GET | /store-loans | รายการสินเชื่อร้าน |
| POST | /store-loans | สร้างสัญญาสินเชื่อร้าน |
| GET | /store-loans/:id | รายละเอียดสัญญา |
| PUT | /store-loans/:id | แก้ไขสัญญา (active เท่านั้น) |
| PATCH | /store-loans/:id/cancel | ยกเลิกสัญญา |
| GET | /store-loans/:id/calculate | คำนวณ PMT (รับ principal, rate, installments → คืน monthly_payment) |
| POST | /store-loans/:id/payments | บันทึกการชำระงวด |
| GET | /store-loans/:id/payments | ประวัติการชำระงวดทั้งหมด |

## กฎสำคัญ

| กฎ | รายละเอียด |
|----|-----------|
| Elasticsearch ค้นหา | ห้ามใช้ MySQL LIKE — ต้อง index เข้า ES ทันทีเมื่อ create/update |
| ค้นแยก 2 key | เบอร์โทร และ บัตรปชช. เป็น index แยก — ค้นได้อิสระ |
| ค้นผู้ค้ำแยก | guarantor_phone/id_card มี index แยก — ค้นว่าเคยค้ำให้ใครได้ |
| บันทึกข้อมูลตรง | applicant_name/phone/id_card เก็บตรงใน record — ไม่ join customers → ป้องกันข้อมูลหายถ้าลูกค้าถูกลบ |
| ไม่บังคับ customer | loan_application ไม่บังคับผูก customer — รองรับลูกค้าขอข้ามร้าน/ไม่มีในระบบ |
| PMT auto | GET /store-loans/:id/calculate รับ principal, rate, installments → คืน monthly_payment |
| Overdue Job | Job รายวัน 01:00: งวดถึงกำหนดแต่ไม่มี payment → status=overdue + notification |
| completed อัตโนมัติ | ชำระครบงวดสุดท้าย → ระบบเปลี่ยน status=completed ทันที |
| Search รวม | /loans/search คืน as_applicant + as_guarantor + store_loans ใน response เดียว |
